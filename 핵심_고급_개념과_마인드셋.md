# 핵심 고급 개념과 마인드셋

매우 날카로운 질문입니다. '무엇을 할 것인가(What)'를 설계하는 단계가 끝났다면, 이제는 **'어떻게 효율적으로 구축하고 유지할 것인가(How)'**에 대한 심화 지식을 습득해야 합니다.

도트게임 제작이라는 특성과 말씀해주신 백엔드/프론트엔드 조합을 고려했을 때, 개발 과정에서 반드시 알아두면 프로젝트의 안정성과 확장성 측면에서 큰 도움이 될 **5가지 핵심 고급 개념과 마인드셋**을 정리해 드립니다.

---

## **🧠 I. 시스템 아키텍처 심화: 데이터 흐름 관리 (Data Flow)**

### **1. 상태 불변성 원칙 (Immutability Principle)**

이것은 가장 중요한 백엔드/게임 로직의 원칙입니다.

- **개념:** 한 번 결정된 값이나 상태는 절대 임의로 수정되어서는 안 되며, 새로운 값이 필요할 때는 **항상 완전히 새로운 객체(New State)**를 생성하여 대체해야 합니다.
- **왜 중요한가?:** 게임은 '현재 시점'이라는 것이 매우 중요합니다. 만약 여러 에이전트가 동시에 같은 데이터를 덮어쓰려 할 때 충돌(Race Condition)이 발생하는데, 상태 불변성을 지키면 누가 어떤 값을 최종적으로 결정했는지 추적하기 쉬워져 버그를 잡기가 압도적으로 쉬워집니다.
- **활용:** Java에서 엔티티를 수정할 때 `save()` 대신 **새로운 DTO(Data Transfer Object)**에 데이터를 담아 새로운 객체로 처리하는 습관을 들이세요.

### **2. 이벤트 기반 아키텍처 (Event-Driven Architecture, EDA)**

현재의 시스템은 요청/응답(Request/Response) 방식이 주를 이룹니다. 하지만 실시간 게임은 **'사건 발생'**에 초점을 맞춰야 합니다.

- **개념:** "누가 무엇을 했다"라는 사실 자체(Event)를 중심 잡고 시스템을 설계하는 것입니다. (예: `PlayerA_Moved`, `ItemB_Collected`, `Combat_Started`)
- **구현 원칙:** 핵심 로직은 요청(`POST /move`) 처리 대신 **이벤트 리스너**가 되어야 합니다. 예를 들어, "움직임 이벤트(Event)"가 서버에 도착하면 $\rightarrow$ 물리 엔진 에이전트가 이벤트를 받아 충돌 판정을 하고 $\rightarrow$ 최종 결과를 새로운 "위치 업데이트 이벤트"로 발행하는 식으로 흐름을 만드세요.
- **효과:** 시스템의 각 부분이 독립적으로 동작하며, 한 모듈에 문제가 생겨도 다른 모듈까지 전부 멈추지 않고 어느 부분에서 문제가 시작되었는지 추적이 매우 용이해집니다.

## **✨ II. 프론트엔드 및 게임 구현 심화 (Client Side)**

### **3. 전역 상태 관리와 컴포넌트 분리의 최적화**

React 자체의 Context API나 Redux 등 상태 관리를 사용하겠지만, 일반적인 웹 페이지 상태관리 방식과 **게임 상태관리**는 다릅니다.

- **Rule:** "어떤 데이터(좌표, 체력)가 변경되었는지"만 관리하고, **데이터를 화면에 그리는 로직(Rendering)**은 최대한 분리해야 합니다.
- **팁: Canvas와 React의 조합을 신중히 하세요.** 전체 게임 렌더링 루프는 `requestAnimationFrame`과 같은 브라우저 네이티브 타이밍 함수를 이용해 제어하고, 이 결과를 주기적으로 가져와서 (혹은 전용 훅을 통해) 특정 React 컴포넌트 내부에 "최종적으로 그려야 할 좌표 목록"으로 바인딩하는 것이 가장 안정적입니다.

### **4. 시간과 속도 관리의 분리 (Time and Speed Separation)**

게임 로직에서 `시간` 개념은 매우 중요하며 까다롭습니다.

- **문제점:** 웹 브라우저나 서버는 '실제 경과 시간'을 기준으로 동작합니다. 하지만 게임에서는 "턴(Turn)" 또는 "틱(Tick)"이라는 가상의 시간이 필요할 때가 많습니다.
- **해결책:** 물리 엔진이나 규칙 기반 로직은 **실제 시간 대신, 일정 간격(`Delta Time`)으로 작동하는 내부 '게임 틱'을 기준으로 연산**하도록 설계해야 합니다. (예: `deltaTime` = 지난 프레임 이후 경과된 초). 이렇게 해야 플레이어가 브라우저를 잠시 비활성화했다가 돌아와도 게임의 물리 법칙이 깨지지 않습니다.

## **🚀 III. 개발자 마인드셋 및 프로세스 관리 (The Mindset)**

### **5. 테스트 주도 개발(TDD) 원칙을 핵심 로직에 적용하기**

개발 초기에 가장 시간을 많이 잡아먹는 것이 "기획과 구현의 싸움"입니다. 이 경우, 기획 단계가 끝난 후 **'테스트 코드 작성'**으로 바로 넘어가세요.

- **원칙:** 기능을 개발하는 순서가 `(1) 테스트 실패 예측 $\rightarrow$ (2) 통과할 수 있는 최소한의 코드를 작성 $\rightarrow$ (3) 기능 완성 및 리팩토링` 입니다.
- **적용 지점:** 복잡한 게임 규칙, 예를 들어 **'이 공격은 피격 시 50% 확률로 스턴(Stun) 상태가 되게 한다'**와 같은 조건문이나, **'캐릭터 A가 캐릭터 B를 피해갈 때의 충돌 판정 공식'**처럼 논리적 검증이 필요한 부분에 TDD를 적용하면 아키텍처가 훨씬 견고해집니다.

### **📚 요약: 개발 순서 재조정**

| **단계** | **목표 (이번 주에 반드시 끝낼 것)** | **초점 기술/개념** |
| --- | --- | --- |
| **1단계 (MVP)** | 움직임만 구현하기 (돌아다니는 점 두 개). | WebSocket, Spring Boot API, Canvas API. |
| **2단계** | 규칙 추가하기 (충돌 감지). | Physics Engine 역할 분리, `Delta Time` 개념 도입. |
| **3단계** | 상태 저장 및 복구 (로그인 후 다시 접속해도 같은 곳에서 시작). | JPA 엔티티 설계, 트랜잭션 관리(Isolation). |

이러한 고급 지식들은 "지식"이라기보다는 "사고방식"에 가깝습니다. 이 개념들을 염두에 두고 코드를 짜게 되면, 나중에 생길 복잡한 버그의 원인과 해결책을 훨씬 예측할 수 있게 될 것입니다.
