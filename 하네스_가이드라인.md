# 하네스 가이드라인

문의하신 **Antigravity 2.0 환경의 하네스 프로세스(Harness Process)** 설정 방법은 해당 시스템이 구체적으로 어떤 목적(예: 테스트 자동화, 배포 파이프라인 관리, 시뮬레이션 등)으로 설계되었는지에 따라 접근 방식이 매우 달라집니다.

**Antigravity 2.0**이라는 환경 자체가 특정 기업이나 전문 분야에서 사용하는 내부 프레임워크일 가능성이 높으므로, 만약 제가 정확한 매뉴얼을 갖고 있지 않다면 일반적인 최고 수준의 시스템 엔지니어링 원칙에 따라 가장 포괄적이고 체계적인 설정 가이드를 제공해 드리겠습니다.

---

### **📝 하네스 프로세스 설정의 단계별 개념 가이드 (Conceptual Guide)**

하네스 프로세스는 일반적으로 **핵심 테스트나 로직을 실행하기 위한 격리된 환경(Sandbox)** 또는 **실행 환경 전체를 관리하는 상위 워커(Orchestrator)** 역할을 합니다. 이 설정을 성공적으로 완료하려면 다음 5단계를 따라야 합니다.

#### **Step 1: 사전 준비 및 요구 사항 정의 (Pre-requisites & Definition)**

가장 먼저, 하네스가 무엇을 감싸서 실행할지, 그리고 그 실행에 필요한 모든 자원을 파악해야 합니다.

- **① 목표 명확화 (Define Scope):** 이 하네스가 테스트 케이스를 실행하는 것인지, 아니면 배포 환경 전체의 무결성을 검증하는 것인지 목적을 정의합니다.
- **② 의존성 매핑 (Dependency Mapping):** 프로세스 실행에 필요한 모든 외부 라이브러리, 데이터베이스 연결 정보, API 키 등을 리스트업 합니다. (예: `Module A`가 작동하려면 `Library B v2.1`이 필요함)
- **③ 환경 변수 정의:** 하네스가 접근해야 하는 모든 시스템 레벨의 설정값(IP 주소, 포트 번호, 인증 토큰 등)을 모읍니다.

#### **Step 2: 핵심 구성 파일 작성 (Configuration File Creation)**

대부분의 최신 시스템은 코드가 아닌 설정 파일을 통해 프로세스를 정의합니다. YAML 또는 JSON 형식을 사용하는 것이 일반적입니다.

- **`harness_config.yaml` 예시:**
    
    ```yaml
    # 1. 환경 지정
    environment: "Staging" # 개발/스테이징/운영 중 어디에서 실행할지
    
    # 2. 프로세스 정의 및 순서
    processes:
      - name: "InitializationCheck"
        module: "./src/checks/init_check.py"
        timeout: 60s # 최대 대기 시간 설정
        depends_on: [] # 선행 작업 없음
        
      - name: "CoreTestRun"
        module: "./src/tests/core_test_suite.py"
        timeout: 300s
        # 이 테스트는 InitializationCheck가 성공해야 실행됨을 명시
        depends_on: ["InitializationCheck"]
    
    # 3. 자격 증명 관리 (보안 강화 영역)
    credentials:
      db_connection:
        type: "vault" # 비밀번호를 코드에 직접 쓰지 않고 Vault 같은 시스템에서 가져오도록 설정
        key: "secret/test_api/db_creds"
    ```
    

#### **Step 3: 실행 환경 설정 및 격리 (Environment Setup & Isolation)**

하네스가 안정적으로 작동하려면, 본래 시스템의 동작에 영향을 주지 않는 독립적이고 예측 가능한 공간에서 실행되어야 합니다.

- **① 컨테이너화 (Containerization):** Docker 또는 Kubernetes를 사용하여 하네스 프로세스와 그 의존성을 하나의 배포 단위로 묶습니다. 이는 "환경 불일치 문제(It works on my machine)"를 근본적으로 해결해줍니다.
- **② 자원 할당:** CPU, 메모리, 네트워크 대역폭 등 필요한 최소한의 시스템 자원을 명시적으로 하네스 컨테이너에 할당합니다. (`resource limits` 설정).

#### **Step 4: 실행 및 검증 (Execution and Validation)**

프로세스를 실제로 구동하고, 예상대로 움직이는지 철저히 테스트해야 합니다.

- **① Dry Run 테스트:** 실제 데이터나 자원에 접근하기 전에, 하네스가 성공적으로 실행되는 워크플로우만 순차적으로 돌려봅니다.
- **② 예외 처리 로직 구현 (Error Handling):** 프로세스 중간에 실패가 발생했을 때(예: DB 연결 끊김), 시스템이 강하게 종료되지 않고, 어떤 오류 코드와 로그를 남기며 사용자에게 알림을 줄지 정의합니다. (`try-catch` 메커니즘).
- **③ 로깅 구조 설계:** 모든 단계의 시작, 성공 여부, 실패 원인 등을 표준화된 형식(JSON 등)으로 기록하는 중앙 집중식 로거를 설정합니다.

#### **Step 5: 모니터링 및 최적화 (Monitoring & Optimization)**

하네스가 안정적으로 작동하더라도 주기적인 점검은 필수입니다.

- **① 성능 프로파일링:** 실행 시간이 너무 길거나 자원 사용량이 비정상적으로 높지 않은지 측정하고, 병목 지점(Bottleneck)을 찾아 최적화합니다.
- **② 알림 시스템 연동 (Alerting):** 하네스 프로세스가 실패하거나 오랜 시간 동안 멈춰 있을 경우, 자동으로 개발팀이나 운영팀에게 Slack/PagerDuty 등을 통해 경고를 보내도록 설정합니다.

### **💡 요약 체크리스트 (Summary Checklist)**

| **항목** | **설명** | **확인 방법** |
| --- | --- | --- |
| **정의** | 하네스의 역할과 범위를 명확히 파악했는가? | 테스트 케이스 목록, 실행 순서도 작성 |
| **환경** | 모든 의존성과 자원(DB, API 등)을 격리된 컨테이너에 담았는가? | Dockerfile/k8s Manifest 확인 |
| **설정** | 프로세스 흐름과 시간 제한(`timeout`)이 설정되었는가? | YAML/Config 파일 검토 |
| **보안** | 민감한 정보(비밀번호, 키)를 코드에 직접 넣지 않고 관리하는가? | Secret Manager (Vault 등) 사용 확인 |
| **검증** | 예외 상황 발생 시의 처리 로직과 로그 기록이 갖추어져 있는가? | 에러 핸들링 테스트 수행 |
